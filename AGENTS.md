# AGENTS.md

## Working Memory

`.scratchpad/` は、AIエージェントがセッションをまたいで作業状況、設計判断、失敗事例、実行環境固有の問題を引き継ぐための永続的なWorking Memoryです。Git管理対象であり、作業ログやGit履歴の代わりではありません。

実質的な作業を始める前に、必ず `.scratchpad/current.md` を読みます。次のファイルは、現在のタスクに関係する場合のみ読みます。毎回すべてを無条件に読み込まないでください。

- `.scratchpad/decisions.md`
- `.scratchpad/lessons.md`
- `.scratchpad/quirks.md`
- `.scratchpad/backlog.md`

作業完了時は、Working Memoryを次のように整理します。

1. `completed.md` が存在しなければ作成し、完了内容を一行または二行で記録する。
2. 設計上の理由を `decisions.md` に残す。
3. 再利用可能な知見を `lessons.md` に残す。
4. 実行環境固有の問題や回避策を `quirks.md` に残す。
5. 完了した内容を `current.md` から削除する。
6. 対応済みの項目を `backlog.md` から削除する。

---

## Prompt Files

`.github/prompts/` には、Working Memoryの更新やタスク完了処理など、再利用可能なプロジェクト固有のPromptがあります。

現在のタスクと名前または目的が一致するPromptが存在する場合は、手順を即興で組み立てる前にそのPromptを優先して使用します。Promptに含まれるプロジェクト固有の規約や手順を、独自に再実装しないでください。

---

## Communication Style

回答は簡潔にします。

- 結論と変更内容を先に伝える。
- 不要な長文説明を避ける。
- Gitやプロジェクト構造など、保守担当者が既に理解している一般知識を説明しない。
- 一つの変更だけを行った場合、その一つだけを説明する。
- 作業結果、確認結果、残っている問題を明確に区別する。
- 問題がなければ、過剰な見出しや報告書形式を使わない。

---

## Scope Control

依頼された範囲以外は変更しません。

関連する改善案があっても、自動では実施しません。必要な場合は提案のみ行います。

リファクタリングやフォーマット変更は、依頼内容の達成に必要な範囲へ限定します。

---

## Constructive Pushback

**依頼内容に明確な懸念がある場合のみ**、無条件に実装せず、原則一文または二文で短く指摘します。

対象には、不要な抽象化、非推奨APIや古い実装方式、より単純な代替案、将来の保守上の問題、セキュリティ上の危険、データ消失や互換性破壊の可能性、意図しない副作用が起こりやすい実装を含めます。

軽微で自明な変更については、不要な反論をしません。

---

## Secrets Hygiene

ファイル、ログ、テストデータ、fixture、生成物、Prompt、コミット対象へ書き込む前に、秘密情報が含まれないか確認します。

次の情報は平文で保存しません。

- APIキー
- アクセストークン
- パスワード
- 秘密鍵
- Cookie
- 認証ヘッダー
- セッション情報
- 個人情報を含む実データ
- 本番環境の認証情報

秘密情報が必要な場合は、環境変数、Secret Manager、暗号化された設定、既存の安全な認証方式を使用します。

サンプル値には、実在しないことが明確な値（`YOUR_API_KEY`、`example-token`、`test@example.invalid` など）だけを使用します。

---

## Comments

既存コメントは原則として保持します。

削除できるのは、コメント対象のコード自体を削除する場合、コードの意味が変更されコメントが事実と一致しなくなった場合、またはユーザーから明示的に削除を依頼された場合だけです。

コードを変更したという理由だけで関連コメントを削除しません。

新しいコメントやDocstringは無闇に追加しません。

ユーザーから依頼された場合、マジックナンバーや回避策の理由を説明する場合、コードだけでは意図を理解しにくい場合、将来の変更で壊れやすい制約を示す場合に限って追加します。

コードの内容をそのまま言い換えるコメントは禁止します。

---

## Lint Warnings

Lint警告を隠すためだけの変更を禁止します。

`TODO`を`@TODO`などへ変更する、変数名を不自然に変更する、不要な整形を行う、理由なくLint無効化コメントを追加する、警告対象コードを別ファイルへ移す、といった対応をしません。

新しいLint警告を発生させた場合は修正するか、修正できない理由を作業結果で報告します。

既存のLint警告を無断で抑制しません。

---

## File Operations

現在の実行環境に差分を確認しやすい構造化ファイル編集ツールがある場合は、それを優先します。

VS CodeまたはCopilot環境では、利用可能な場合に `create_file`、`replace_string_in_file`、`multi_replace_string_in_file` を使用します。

一括置換では変更前後の文脈を明示し、完全一致する範囲だけを対象にします。

ソースコードの編集に `sed`、`perl -i`、`cat` によるheredoc書き込み、インラインPythonスクリプト、インラインNode.jsスクリプト、広範囲な正規表現置換を安易に使用しません。

大規模で機械的な変更に限りスクリプトを使用できますが、次の条件をすべて満たします。

1. 変換規則が決定的である。
2. 対象ファイルが明確に限定されている。
3. 実行前に変更対象を確認できる。
4. 実行後にフォーマッター、Lint、テストを実行する。
5. 生成された差分を確認する。
6. 意図しない変更がないことを確認する。

ファイル編集後は、変更箇所だけでなく、周辺の構文や構造が壊れていないことも確認します。
